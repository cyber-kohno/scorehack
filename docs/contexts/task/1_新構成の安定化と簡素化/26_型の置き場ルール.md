# 型の置き場ルール

## 目的
新構成で型の置き場が再び散らばらないように、まず最小限のルールを決める。
特に、
- `domain/*`
- `state/*-store.ts`
- `*-store.ts` の namespace façade
の使い分けを明確にする。

## 基本ルール
### 1. 業務概念の型は `domain/*` に置く
例:
- melody note
- outline element
- arrange relation
- playback option

判断基準:
- store がなくても意味を持つ
- app / ui / state をまたいで使う
- 「何を扱っているか」の言葉そのものになっている

### 2. store 専用の状態型は `state/*-store.ts` に同居させる
例:
- project file handle state
- terminal state
- preview state
- focus state

判断基準:
- その store のためだけに存在する
- state shape / initial value / writable が密接に結びつく
- 他の層で単独の業務概念としては使わない

### 3. namespace façade は互換性か集約価値があるときだけ残す
例:
- `melody-store.ts`
- `arrange-store.ts`
- `piano-editor-store.ts`

判断基準:
- 既存 caller が多く、急に外すと影響が広い
- 型と関数の集約窓口として読む価値がある
- 単なる別名の束にしかなっていないなら、将来的な削除候補

## 今回の小さい適用例
- `project-file-store.ts` は store 専用 state なので、namespace をやめて
  - `ProjectFileHandle`
  - `ProjectFileState`
  - `INITIAL_PROJECT_FILE_STATE`
  を同じファイルで直接 export する形にした。

## 当面の方針
- `state/*-store.ts` の namespace は、薄いものから順に平らにできるか見る
- `domain/*-store.ts` の façade は、いったん残してよい
- ただし、`melody-store.ts` のような façade は今後
  - 本当に集約窓口として価値があるか
  - `melody-types.ts` へ直接寄せた方がよいか
  を個別に見直す
