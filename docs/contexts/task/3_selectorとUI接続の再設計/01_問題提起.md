# 問題提起

## 背景

現状の `createMelodySelectors` は `RootStoreToken` を受け取っているが、内部でたどる先では `get(...)` や dedicated store の直接参照が多く、`RootStoreToken` は実質的な依存を表せていない。

そのため、見た目の API と実際の依存がずれている。

さらに、現在の動作は `src/state/root-store.ts` の広い invalidation に支えられている部分があり、selector 単体としては「何の変更で再評価されるべきか」が曖昧になっている。

## 今回の前提

このタスクでは、`createXxxYyy(...)` が受け取るものを次のように定義する。

- `RootStoreToken` ではなく、caller 側が持っている **必要な最新値**
- その最新値は、UI コンポーネントであれば `$store` から reactive に渡す

例:

```ts
$: melodySelectors = createMelodySelectors($modeStore, $melodyCursorStore);
```

この方針を取る理由は、selector 内で `get(store)` を行っても、caller 側の reactive 依存としては検知されないためである。

## このタスクで気をつける点

`createMelodySelectors` の caller は UI コンポーネントだけではない。
現時点では少なくとも次のような business logic 側からも使われている。

- `melody-input`
- `melody-sync-actions`
- `melody-focus-sync-actions`
- `preview-util`
- `terminal-reducer`
- `terminal/sector/builderMelody`

そのため、このタスクの実態は「selector と UI 接続」だけでなく、**selector と caller 接続全体の再設計** である。

ただし、横展開はまだ行わず、まずは `createMelodySelectors` のみを対象に思想を固める。
