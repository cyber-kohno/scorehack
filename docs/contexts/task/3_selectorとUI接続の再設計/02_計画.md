# 計画

## ゴール

`createMelodySelectors` を、`RootStoreToken` を受ける曖昧な selector factory から、**必要な最新値だけを受け取る明確な selector factory** へ再設計する。

## 進め方

### 1. 現状依存の棚卸し

まず `createMelodySelectors` が提供している API ごとに、

- 本当に何の値に依存しているか
- `RootStoreToken` を経由しているだけで実際には不要な依存は何か

を洗い出す。

対象 API:

- `getCurrentScoreTrackOrThrow`
- `getCurrentFocusRange`
- `isCurrentCursorOverlapping`
- `findNearFocusIndex`

### 2. selector の入力を設計し直す

次に、上記 API を支える入力を `RootStoreToken` ではなく、

- current score track
- current focus range
- current cursor

のような **必要な最新値** に置き換える設計を決める。

ここでは次の点も判断する。

- `createMelodySelectors` を 1 つのまま維持するか
- 依存が多すぎる場合、より小さい selector factory に分けるべきか

### 3. caller 接続を修正する

`createMelodySelectors` を呼んでいる各 caller を、新しい入力方式へ追従させる。

優先順位:

1. `melody` 内 caller
2. `playback` / `terminal` など melody 外 caller

### 4. 動作確認

少なくとも以下を確認する。

- `npm run check`
- `npm run build`
- `cargo check`

必要に応じて、melody の基本操作も手動確認対象に含める。

## 今回やらないこと

- `outline` への横展開
- `commit()` の selective update 拡張
- `root-store` invalidation の撤去
- selector 以外の参照系 utility の全面整理

今回は `createMelodySelectors` を練習台として、参照系 utility の設計指針を固めるところまでに留める。
