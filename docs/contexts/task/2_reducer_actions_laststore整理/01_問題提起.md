# 問題提起

## 背景
- レガシーでは `Reducer`, `Store`, `Input` などが広い責務を持っていた
- `lastStore` を受け取って、更新と参照の両方を行う構造になっていた
- しかし新構成では state 実体が dedicated store に分割されている

## 今の違和感
- 更新専用 helper にまで `RootStoreToken` が残っている
- `reducer` の中に
  - mutation
  - selector/accessor
  - orchestration
  が混在している
- `actions` の一部は reducer の薄い façade に見える

## なぜ問題か
- 「何が更新責務なのか」が読み取りにくい
- 「何が表示用の読み取りなのか」が reducer の中に埋もれる
- `RootStoreToken` が本当に必要な場所と、慣性で残っている場所が区別しにくい
- 今後 helper 名や接頭辞を決めても、責務整理が済んでいないと再び揺れる

## 仮説
- 更新専用 helper は、かなりの範囲で `RootStoreToken` を不要にできる
- 本当に `RootStoreToken` が必要なのは
  - project-data / cache の合成
  - 表示用 selector
  - shell / playback / keyboard などの orchestration
  に寄るはず
- そのため、先に `reducer` の責務を分解するのが筋
