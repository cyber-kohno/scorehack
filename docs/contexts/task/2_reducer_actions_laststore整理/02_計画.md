# 計画

## 目標
- `reducer` という塊を、そのまま温存せずに責務単位へ分解する
- `RootStoreToken` が不要な更新専用 helper を見えるようにする
- その結果として、`actions` や `reducer` の命名を後から自然に決められる状態にする

## 進め方

### 1. 現状棚卸し
- 主要な `reducer` / `actions` を一覧化する
- それぞれを
  - mutation
  - selector/accessor
  - orchestration
  に分類する

### 2. 最初の対象を決める
- 最初は `melody-reducer` か `outline-reducer` のどちらかに絞る
- 条件:
  - caller が多すぎない
  - mutation と selector の混在がはっきりしている
  - 小さい分離で価値が出る

### 3. 小さい分離を行う
- まず mutation だけを小さく外す
- 次に selector/accessor を外す
- orchestration は最後に残して扱う

### 4. 命名判断は後段に回す
- `actions` を残すか
- `reducer` を rename するか
- 接頭辞をどう付けるか

これらは、責務分解の後に判断する

## 当面の優先順
1. `melody-reducer` の責務棚卸し
2. `outline-reducer` の責務棚卸し
3. 小さい mutation 群の `RootStoreToken` 依存見直し
4. `actions` の pass-through 性評価

## token 軸の作業順
1. token 不要な関数を factory から外す
2. token が必要な関数群を factory 化する
3. token 不要な関数群を namespace でまとめるか判断する

補足:
- 3 は今時点では優先度を下げる
- まず 1 と 2 を進める

## 完了の目安
- 少なくとも 1 feature で
  - mutation
  - selector/accessor
  - orchestration
  の分離方針が形になっていること
- その時点で、命名や import 規約を決める土台ができる
