# Phase 12 Next Phase Options

## 目的
このドキュメントは、Phase 12 時点で
次にどの方向へ進むのが自然かを比較するためのメモです。

---

## 候補 1: `cache` selector / accessor を先に増やす

### 内容
- `cache` を直接読む component / reducer をすぐ書き換えるのではなく、
  まず読み取り入口を整理する

### 良い点
- 既存動作を壊しにくい
- `cache` の責務分解と相性がよい
- 段階移行しやすい

### リスク
- 再計算入口そのものの整理は後ろに残る

### 判断
- 最も安全な選択肢

---

## 候補 2: `reducerCache.ts` の入口整理を先に進める

### 内容
- calculate の呼び出し契機や再計算入口を `app` / `state` 側に寄せる

### 良い点
- `data -> cache` の更新契機が見やすくなる
- 将来の分割に効く

### リスク
- 影響範囲が広い
- いきなり進めると操作後の再計算漏れが怖い

### 判断
- 有力だが、selector 整理よりは重い

---

## 候補 3: `cache` の物理分割に進む

### 内容
- `store.ts` から `cache` 実体を独立させる

### 良い点
- 派生 state としての境界はきれいになる

### リスク
- 現時点では依存先がまだ広い
- 再計算入口整理が不十分なまま進むことになる

### 判断
- まだ早い

---

## 候補 4: `data` の物理分割に戻る

### 内容
- `cache` 整理を止めて `data` 分割へ進む

### 良い点
- project data 境界を先に確定できる

### リスク
- `cache` が最大のリスク源として残る
- いま戻ると理解が浅いまま分割へ入る形になる

### 判断
- 今は推奨しない

---

## 現時点の推奨
次フェーズは

1. `cache` selector / accessor の整理
2. その後に `reducerCache` の入口整理

の順が最も自然。

理由:
- 影響範囲の大きい `cache` を小さく扱い始められる
- `cache` の責務分解と相性がよい
- いきなり再計算契機へ踏み込むより安全

---

## 結論
次フェーズは、
`cache の読み取り入口整理`
を主要対象にするのが最も安全。
