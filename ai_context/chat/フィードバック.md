現行 `refer_app` の解析結果です。結論からいうと、これは「Svelte製UI」よりも、**作曲設計専用の独自エディタ基盤**としてかなり作り込まれていて、すでに仕様の核は十分入っています。再構築の価値もかなり高いです。

**全体像**
起点は [Entry.svelte](C:/app/git/scorehack/refer_app/src/Entry.svelte) で、起動時にキャッシュ計算とレイアウト初期化を行い、グローバルなキーボード入力を [inputRoot.ts](C:/app/git/scorehack/refer_app/src/system/input/inputRoot.ts) に集約しています。画面は [MainFrame.svelte](C:/app/git/scorehack/refer_app/src/system/MainFrame.svelte) で、`Outline / Timeline / Terminal / ArrangeEditor / ArrangeFinder` を状態に応じて切り替える構造です。

状態管理は [store.ts](C:/app/git/scorehack/refer_app/src/system/store/store.ts) の単一 store に集約されていますが、実際の更新は Svelte 的な immutable 更新ではなく、`lastStore` を reducer 群が直接書き換えて `commit()` する方式です。つまり設計思想としては「Svelte UI + 独自 reducer/state machine」です。

**現時点で実装されている仕様**
楽曲の中心データは [storeOutline.ts](C:/app/git/scorehack/refer_app/src/system/store/props/storeOutline.ts) の `elements` です。ここに `init / section / chord / modulate / tempo / ts` の時系列要素を並べ、楽曲構成と進行を表現しています。これがまさに設計フェーズの核です。

そこから [reducerCache.ts](C:/app/git/scorehack/refer_app/src/system/store/reducer/reducerCache.ts) が、
- 小節・拍・時間
- コード位置
- 転調/テンポ変化の反映
- タイムライン描画位置
- アレンジ紐付け
をキャッシュ展開しています。  
このキャッシュを前提に、アウトライン表示、タイムライン表示、再生位置同期まで動いています。

入力面はかなり明確です。
- ハーモニー編集: [inputOutline.ts](C:/app/git/scorehack/refer_app/src/system/input/inputOutline.ts)
- メロディ編集: [inputMelody.ts](C:/app/git/scorehack/refer_app/src/system/input/inputMelody.ts)
- ターミナル操作: [inputTerminal.ts](C:/app/git/scorehack/refer_app/src/system/input/inputTerminal.ts)

ハーモニー側では、
- セクション追加
- コード追加
- 転調要素追加
- コード degree/symbol/拍/eat 編集
- 範囲選択削除
- アレンジ editor/finder 起動
- 全体プレビュー
まで入っています。

メロディ側では、
- 単旋律ノート入力
- 音高移動
- 音価変更
- ノート長変更
- ノート移動
- 範囲選択
- コピー/ペースト
- スケール内移動
- 連符系操作の一部
まで実装されています。  
「歌メロ単旋律の高速試行錯誤」という目的にちゃんと寄っています。

**ターミナル機能**
ターミナルは単なるログ表示ではなく、**文脈依存コマンドUI**です。[reducerTerminal.ts](C:/app/git/scorehack/refer_app/src/system/store/reducer/reducerTerminal.ts) と [commandRegistUtil.ts](C:/app/git/scorehack/refer_app/src/system/store/reducer/terminal/commandRegistUtil.ts) が、現在位置に応じて使えるコマンドを切り替えています。

確認できた主なコマンド群は、
- 共通: `clear`, `help`, `save`, `load`
- harmonize: `lsh`, `mkh`, `cht`, `sf`, `volume`
- init: `scales`, `tempo`, `scale`
- section: `ren`
- chord: `lsb`
- modulate: `domm`, `key`, `parallel`, `relative`
- melody: `lss`, `lsa`, `mks`, `mka`, `aul`, `rms`, `chs`, `sf`
- piano arrange: `ubk`, `dbk`

です。  
この方向性は、今回の「メニューなし・キーボード完結」という思想と非常に一致しています。

**プレビュー/音まわり**
[previewUtil.ts](C:/app/git/scorehack/refer_app/src/system/util/preview/previewUtil.ts) で、キャッシュから再生タイミングを計算し、`soundfont-player` を使ってノート再生しています。  
対応範囲は、
- メロディトラック再生
- オーディオトラック再生
- ピアノアレンジのパターン再生
- 再生中のフォーカス同期
- 再生ライン表示
です。  
つまり「最終音源」ではなく、**設計確認のための試聴**として設計されています。

**アレンジ機能の位置づけ**
[storeArrange.ts](C:/app/git/scorehack/refer_app/src/system/store/props/arrange/storeArrange.ts) を見ると、アレンジは楽曲本体ではなく、**コード列に対する補助レイヤー**として実装されています。現状メインは piano editor/finder で、`relation` により chordSeq とパターンを紐付けています。  
ここは将来の DTM/AI 連携に向けた「中間表現」としてかなり重要な層です。

**技術的な現状認識**
再構築前提で見たときの大きな論点はこのあたりです。

- Svelteの責務分離が弱く、UIと業務ロジックが密結合
- store が巨大で mutable 更新前提のため、追跡と保守が難しい
- reducer/input/util の境界が揺れている
- キャッシュ依存が強く、仕様変更時の影響範囲が広い
- 命名ゆれや typo が多い (`disignInitializer`, `Termianl`, `Medhod` など)
- 文字化けしたコメントが多く、意図の再読コストが高い
- 一部未整理/壊れたファイルがある  
  例: [ArrangeNote.svelte](C:/app/git/scorehack/refer_app/src/system/component/timeline/grid/arrange/ArrangeNote.svelte)

補足として、`npm.cmd run check` は実行しましたが、sandbox の `spawn EPERM` が大量に出て Svelte style preprocess が正常評価できていません。その制約を除いても、少なくとも [ArrangeNote.svelte](C:/app/git/scorehack/refer_app/src/system/component/timeline/grid/arrange/ArrangeNote.svelte) には実際の型/参照エラーが残っています。

**今回の解析で把握できた仕様の核**
このアプリの本質は、
- 楽曲を「要素列」で持つ
- 要素列から時間・拍・コードをキャッシュ展開する
- その上で harmonize / melody / arrange をキーボードだけで編集する
- プレビューは設計検証用
という構造です。

なので、リプレース時の中心ドメインはまず
- `Song / Element / Section / Chord / Modulation / TempoChange / TimeSignatureChange`
- `MelodyTrack`
- `ArrangeTrack`
- `PreviewProjection`
あたりで再定義するのが自然だと思います。
