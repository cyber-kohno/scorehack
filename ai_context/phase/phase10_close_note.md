# Phase 10 Close Note

## Phase 10 の到達点
Phase 10 では、
`data` の物理分割に入る前に、
いま何が残っていて、どこまで揃えば安全かを判断できる状態を目指した。

このフェーズで整理できたことは次の通り。

- `data` 直参照の残件を再洗い出しした
- 残件を `UI / input / reducer-cache / playback / 正規接点` に分類した
- `data` 分割前チェックリストを作成した
- 次フェーズ候補を比較した

---

## 現時点の判断

### 1. まだ `data` 物理分割へ即着手する段階ではない
理由:
- playback の入口整理が残っている
- legacy component の直参照が残っている
- reducer / cache の扱い方針をもう少し明示したい
- arrange 境界の最終確認も必要

### 2. 次に自然なのは playback / UI 残件の最終整理
理由:
- 既存動作を壊しにくい
- `project-data` 方針をさらに浸透させられる
- `data` 分割前チェックの未完了項目に直接効く

### 3. その後に `cache` を主要対象にする判断が見えてくる
理由:
- `reducerCache.ts` は依然として最重要リスク箇所だから

---

## このフェーズを閉じてよい理由
- `data` 分割の可否判断に必要な材料が揃った
- 次に何をやると安全かを説明できる
- 無理に分割へ入らず、一段整理を挟むべき理由が明確になった

---

## 次フェーズへの引き継ぎ
次は `Phase 11` として、
`playback / UI 残件の最終入口整理`
を主要対象にするのが自然。
