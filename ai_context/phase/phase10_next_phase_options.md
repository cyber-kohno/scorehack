# Phase 10 Next Phase Options

## 目的
このドキュメントは、Phase 10 時点で
次にどの方向へ進むのが自然かを比較するためのメモです。

---

## 候補 1: `data` の物理分割に進む

### 内容
- `store.ts` の `data` を独立した実体として切り出す

### 良い点
- project data 境界がさらに明確になる
- save / load 対象と state 構造が一致する

### リスク
- まだ playback と reducer / cache 側に直接参照が残る
- arrange の境界が十分固まっていない
- 今入ると分割作業そのものが探索的になりやすい

### 判断
- まだ少し早い

---

## 候補 2: playback / UI 残件の入口整理を先に進める

### 内容
- `previewUtil.ts` の project data 入口を整理する
- legacy component の `$store.data` 直参照をさらに減らす

### 良い点
- `data` 分割前の不安要素を減らせる
- 既存動作を壊しにくい
- `project-data` 方針をさらに浸透させられる

### リスク
- 物理分割自体はまだ先送りになる

### 判断
- 最も安全で自然

---

## 候補 3: `cache` を次の主要対象にする

### 内容
- `reducerCache.ts` を中心に、`data -> cache` 境界を先に整理する

### 良い点
- `data` 分割時の大きなリスク源に先に向き合える
- timeline / playback / outline へ波及する重要箇所を押さえられる

### リスク
- 影響範囲が広い
- `project data` 入口整理が中途半端なままだと理解が散る

### 判断
- 有力だが、次の一手としてはやや重い

---

## 候補 4: arrange を次の主要対象にする

### 内容
- `arrange` の persistent/editor 境界を先に整理する

### 良い点
- 長期的には重要

### リスク
- 現時点では複雑で、分割判断の前に扱うには重い

### 判断
- 今は後回しが妥当

---

## 現時点の推奨
次フェーズは

1. playback / UI 残件の入口整理
2. その後に `cache` 境界の整理

の順が最も自然。

理由:
- `data` 分割前チェックリストで不足している条件に直結している
- 既存動作を大きく壊さずに進めやすい
- その後で `data` 分割 or `cache` 先行 の判断をもう一段精度高くできる

---

## 現時点の結論
次フェーズは、
`data` の即時物理分割ではなく、
`playback / UI 残件の最終入口整理フェーズ`
として設計するのが最も安全。
